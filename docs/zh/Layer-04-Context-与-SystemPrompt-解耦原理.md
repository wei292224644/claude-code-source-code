# Layer 04: Context 与 SystemPrompt 解耦原理

> 基于 Claude Code 提示词工程与架构设计分析 · 2026-04-24

## 1. 核心矛盾：纯文本的随机性 vs 缓存的确定性

虽然 `userContext`（项目规则）和 `systemContext`（系统环境）最终都会被拼接进发送给 LLM 的系统提示词中，但在代码逻辑层面，Claude Code 将它们严格解耦。这种设计是为了解决 LLM 缓存机制中的核心矛盾：**只要字符串有一个字节的差异，整个缓存就会失效。**

---

## 2. 为什么不直接写死在 SystemPrompt 中？

### 2.1 强制一致性的“确定性排序”
`userContext` 和 `systemContext` 在代码中以 `Record<string, string>`（键值对）形式存在。
- **问题**：如果直接写在字符串里，每次重新读取环境变量或项目规则时，顺序可能不一致。
- **方案**：系统在拼接前会对所有 Key 进行 **Alphabetical Sort（按字母顺序排序）**。
- **结果**：无论底层数据读取顺序如何，最终生成的提示词文本在字节级上永远保持一致，确保了提示词缓存（Prompt Cache）的哈希匹配。

### 2.2 多代理环境下的“精细化重写”
子代理（Subagent）经常需要修改父代理的环境参数。
- **场景示例**：启动一个工作区隔离代理（Worktree）。
- **解耦优势**：子代理只需通过对象组合（Object Composition）重写特定字段即可：
  ```typescript
  const subagentContext = { ...parentContext, cwd: "/tmp/isolated-worktree" };
  ```
- **对比**：如果环境信息被写死在庞大的 `systemPrompt` 字符串中，子代理必须使用复杂的正则表达式进行搜索替换，这极大地增加了逻辑风险和代码复杂性。

### 2.3 区分“静态逻辑”与“动态元数据”
- **SystemPrompt (逻辑层)**：定义 Claude 的身份、思考模式、工具调用协议。这是系统的“灵魂”，更新频率极低。
- **Context (数据层)**：描述当前是在 Mac 还是 Linux？使用了什么 Shell？项目有哪些特定规矩？这是系统的“外壳”，随环境而变。
- **缓存策略**：这种分离允许 Anthropic API 实现**分段缓存**。静态部分可以放入长期全局缓存，而 Context 部分可以作为会话级短效缓存，从而实现成本最优化。

### 2.4 结构化遥测与调试
- **可追溯性**：当工具调用报错时，遥测系统可以直接记录结构化的 `userContext` 字段。
- **分析效率**：后端数据分析可以直接筛选“在所有 Windows 用户中发生的报错”，而不需要从一段混乱的提示词文本中去解析系统类型。

---

## 3. 注入流程示意图

1.  **数据收集**：读取环境变量、`CLAUDE.md`、OS 类型等。
2.  **结构化存储**：存入 `Record` 对象，作为 `toolUseContext` 的一部分。
3.  **确定性转化**：
    - `Object.keys().sort()`
    - 拼接为 Markdown 格式（如 `# Environment\n- OS: darwin...`）。
4.  **最终组装**：将排序后的文本插入到 `systemPrompt` 的指定占位符位置。

---

## 4. 架构结论

将 Context 与 SystemPrompt 解耦体现了 **“数据与逻辑分离”** 的软件工程原则。通过在代码中维护结构化的上下文，Claude Code 成功地将高度动态的用户环境转化为了字节级稳定的 API 请求，这正是其能够在高频、多代理并发场景下保持低延迟和低成本的技术基石。
