# Claude Code 记忆系统 - 技术分析

## 概述

Claude Code 实现了一个复杂的多层记忆系统，用于在对话之间持久化信息。本文提供 Claude Code v2.1.88 记忆系统的详细技术分析。

---

## 记忆架构

Claude Code 维护**两个独立的记忆子系统**，它们服务不同目的：

| 子系统 | 用途 | 位置 | 加载策略 |
|--------|------|------|----------|
| **Auto Memory (memdir)** | 关于用户/项目的持久化、有类型的记忆 | `~/.claude/projects/<slug>/memory/` | 系统提示词章节（每轮） |
| **CLAUDE.md (nested_memory)** | 项目/用户指令和规则 | 多个位置 | 基于附件，按需加载 |

## 1. Auto Memory 系统 (memdir)

### 1.1 记忆存储结构

```
~/.claude/
└── projects/
    └── <sanitized-git-root>/  ← 项目级别
        └── memory/
            ├── MEMORY.md              # 索引文件（入口）
            ├── user_role.md           # 用户记忆
            ├── feedback_preferences.md # 反馈记忆
            ├── project_context.md    # 项目记忆
            └── reference_links.md     # 引用记忆

~/.claude/                          # 不在 git 仓库时的备用
└── memory/
    ├── MEMORY.md
    ├── user_role.md
    └── ...

# 团队记忆 (feature-gated, TEAMMEM)
~/.claude/projects/<slug>/memory/
└── team/
    ├── MEMORY.md              # 团队索引
    ├── coding-standards.md
    └── project-decisions.md
```

**源码**: `src/memdir/paths.ts:223-235` (`getAutoMemPath()`)

### 1.2 记忆类型（四类分类法）

记忆系统限制为四类，定义在 `src/memdir/memoryTypes.ts`：

```typescript
const MEMORY_TYPES = ['user', 'feedback', 'project', 'reference'] as const
```

| 类型 | 描述 | 范围 |
|------|------|------|
| **user** | 用户的角色、目标、知识、偏好 | 始终私有 |
| **feedback** | 用户给出的关于避免或重复的指导 | 私有或团队 |
| **project** | 进行中的工作、计划、bug、事件 | 私有或团队 |
| **reference** | 指向外部系统的指针 | 通常团队 |

**源码**: `src/memdir/memoryTypes.ts:14-21`

### 1.3 记忆前置格式

每个记忆文件使用前置格式：

```markdown
---
name: 记忆名称
description: 一行描述，用于在未来对话中判断相关性
type: user|feedback|project|reference
---

记忆内容正文。feedback/project 类型请结构化为：
规则/事实，然后 **Why:** 和 **How to apply:** 行。
```

**源码**: `src/memdir/memoryTypes.ts:261-271`

### 1.4 如何保存记忆

系统提示词指导模型执行**两步流程**：

1. 将记忆写入自己的文件，带前置格式
2. 在 `MEMORY.md` 中添加指针（索引）

```
MEMORY.md 是一个索引，不是记忆本身 — 每条目一行：
- [Title](file.md) — 一行钩子
```

**源码**: `src/memdir/memdir.ts:199-266` (`buildMemoryLines()`)

### 1.5 记忆加载: `loadMemoryPrompt()`

核心函数 `loadMemoryPrompt()` (第419行) 返回系统提示词的记忆指令：

```typescript
export async function loadMemoryPrompt(): Promise<string | null> {
  const autoEnabled = isAutoMemoryEnabled()
  // ...
  if (autoEnabled) {
    const autoDir = getAutoMemPath()
    await ensureMemoryDirExists(autoDir)
    return buildMemoryLines('auto memory', autoDir, extraGuidelines).join('\n')
  }
  return null
}
```

**源码**: `src/memdir/memdir.ts:419-507`


**源码**: `src/constants/prompts.ts:491-495`

## 2. CLAUDE.md 系统 (nested_memory)

### 2.1 文件发现顺序

文件按**逆优先级顺序**加载（最后加载的最高优先级）：

```
1. Managed memory        → /etc/claude-code/CLAUDE.md (全局，始终加载)
2. User memory         → ~/.claude/CLAUDE.md (私有全局)
3. Project memory      → ./CLAUDE.md, .claude/CLAUDE.md, .claude/rules/*.md
4. Local memory       → ./CLAUDE.local.md (私有项目级)
5. Memdir 入口  → ~/.claude/projects/.../memory/MEMORY.md
```

**源码**: `src/utils/claudemd.ts:1-26`

### 2.4 `@include` 指令支持

记忆文件可以包含其他文件：

```
语法: @path, @./relative/path, @~/home/path, @/absolute/path

示例:

@./SPEC.md
@~/notes/architecture.md
```

- 仅在叶子文本节点中工作（不在代码块内）
- 通过 `processedPaths` 追踪防止循环引用
- 不存在的文件被静默忽略

**源码**: `src/utils/claudemd.ts:18-25`

## 3. 记忆加载流程

### 3.1 每轮: 系统提示词记忆章节

```
用户输入
    ↓
QueryEngine.submitMessage()
    ↓
getSystemPrompt() → loadMemoryPrompt()
    ↓
系统提示词包含:
# Memory
You have a persistent, file-based memory system at...
## Types of memory
...
## How to save memories
...
## Memory.md
[索引内容]
```

**每次 API 调用都发生** — 但只有索引（MEMORY.md），不是完整文件内容。

### 3.2 按需: nested_memory 附件

```
模型调用工具（如 Read, Edit）
    ↓
getAttachmentMessages() → getNestedMemoryAttachmentsForFile()
    ↓
检查 loadedNestedMemoryPaths（去重）
    ↓
memoryFilesToAttachments()
    ↓
附件添加到 API 请求
    ↓
模型看到记忆内容
```

**源码**: `src/utils/attachments.ts:1792-1850`

### 3.3 去重: `loadedNestedMemoryPaths`

为防止重复注入相同的记忆文件：

```typescript
if (toolUseContext.loadedNestedMemoryPaths?.has(memoryFile.path)) {
  continue  // 已经加载，跳过
}
toolUseContext.loadedNestedMemoryPaths?.add(memoryFile.path)
```

结合 `readFileState` LRU 缓存进行跨函数去重。

**源码**: `src/utils/attachments.ts:1718-1724`

```
- 何时使用/更新计划而非记忆：
```

## 8. 记忆完整性：什么不应该保存

提示词明确排除：

```
## What NOT to save in memory

- 代码模式、约定、架构、文件路径 — 可从代码推导
- Git 历史、最近的更改 — 使用 `git log` / `git blame`
- 调试解决方案 — 修复在代码中，提交消息有上下文

- 临时任务细节（进行中的工作、临时状态）
```

即使用户明确要求保存 PR 活动日志，模型也被指示询问其中的*令人惊讶/非显而易见的*部分。

## 9. 记忆漂移处理

记忆可能变得过时。系统指示：

```
## Before recommending from memory

命名函数/文件/标志的记忆声称它在该记忆被写入时存在。
它可能已被重命名、删除或从未合并。

在推荐之前：
- 如果命名文件路径：检查文件是否存在
- 如果命名函数或标志：搜索它
- 如果用户要依据它采取行动：先验证

"记忆说 X 存在" ≠ "X 存在"
```

## 12. 关键设计决策

### 12.1 为什么两个记忆系统？

1. **Auto Memory (memdir)**: 类型化、可搜索、跨会话持久化
   - 专为跨对话学习设计
   - 四类分类法防止记忆膨胀
   - MEMORY.md 作为索引防止 token 爆炸

2. **CLAUDE.md**: 直接项目指令
   - 签入版本控制（团队共享）
   - 人类编写，非模型生成
   - 条件规则用于文件特定指导

### 12.2 为什么用附件而非系统提示词？

- 系统提示词有大小限制
- 附件可根据上下文条件加载
- 不同文件可在不同时间加载
- LRU + 去重防止无限记忆增长

### 12.3 为什么用前置格式？

- 支持自动化处理
- 类型字段支持过滤
- 描述支持相关性匹配
- 结构化格式防止漂移
